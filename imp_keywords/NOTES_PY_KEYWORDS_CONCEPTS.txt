NOTES.PY CONCEPTS EXPLAINED
================================================================================

A. Note CRUD Pattern
- CREATE: POST /api/notes/ → New note
- READ: GET /api/notes/ → All user notes
- READ: GET /api/notes/{id} → Single note
- UPDATE: PUT /api/notes/{id} → Edit note
- DELETE: DELETE /api/notes/{id} → Remove note
- PATCH: /star → Toggle star (partial update)

B. User Isolation
- Each user sees only their own notes
- Filter by: Note.user_id == current_user.id
- Prevents viewing other users' notes
- Ownership check on every operation

C. Create Note
- User sends: title, description, color
- Server adds: user_id (from token), timestamps
- Color defaults to "#fff7b1" (yellow) if not provided
- Returns created note with id and timestamps

D. Get All Notes
- Query all notes for current user
- Filter: Note.user_id == current_user.id
- Return: List of all user's notes
- Empty list if user has no notes

E. Get Single Note
- Verify note exists AND belongs to user
- Two conditions: id match AND user_id match
- Return 404 if not found or not owner

F. Update Note
- Only update fields that are provided
- exclude_unset=True = skip unchanged fields
- Example: Update only color, leave title unchanged
- setattr() = dynamic field update

G. Delete Note
- Verify ownership first
- Delete from database
- Return success message
- Note is permanently removed

H. Toggle Star
- PATCH method for partial update
- Toggle is_starred: True ↔ False
- Use: not db_note.is_starred
- Quick favorite/unfavorite functionality

I. Color Storage
- Color stored as hex code: "#fff7b1"
- Can be any CSS color format
- Color persists across sessions
- Updated via PUT endpoint

J. Timestamps
- created_at: When note was created (never changes)
- updated_at: When note was last modified
- Both set automatically by database
- Useful for sorting and filtering
